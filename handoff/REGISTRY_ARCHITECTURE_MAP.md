# Registry Architecture Map

## 1) High-Level Architecture
- Route resolution is hybrid:
- Registry catch-all path: `app/[[...slug]]/page.tsx`
- Section-specific route files also use registry helpers: `app/expertise/page.tsx`, `app/industries/page.tsx`, `app/gallery/page.tsx`
- Dynamic section detail pages (`app/expertise/[slug]/page.tsx`, `app/industries/[slug]/page.tsx`, `app/case-studies/[slug]/page.tsx`) combine content model lookups with registry metadata/template controls.
- Explicit section wrappers under `app/expertise/` and `app/industries/` call `getPageByRoute()` and render templates from registry rows.

Canonical registry chain:
- `src/data/page-registry.csv` -> generated by `scripts/gen-page-registry.ts` -> `src/data/pageRegistry.generated.ts`
- Route lookup via `lib/pageRegistry.ts` or direct scan of `PAGE_REGISTRY`
- Template resolution via `src/templates/registry.ts` (`TEMPLATE_BY_ID` + uploaded fallback)
- Content resolution via `src/content/registry.ts` (`contentByKey`)

## 3) Registry-Backed Page Flow
Flow (canonical):
1. Route request resolves in `app/[[...slug]]/page.tsx` or section route file.
2. Route is normalized and looked up in `PAGE_REGISTRY`/`PAGE_BY_ROUTE`.
3. `templateId` is resolved via `getTemplate()` in `src/templates/registry.ts`.
4. `contentKey` is resolved via `getContentByKey()` in `src/content/registry.ts`.
5. Template component renders with `content`, `pageTitle`, optional `theme`, optional `heroVisualId`.

Notes:
- `getTemplate()` first checks `TEMPLATE_BY_ID`, then `getUploadedTemplate()`.
- `scripts/validate-page-registry.ts` validates template ID resolvability against both legacy + uploaded mappings.

## 4) Gallery Thumbnail Flow
1. Animation metadata is generated into `src/data/animationCatalog.generated.ts`.
2. `src/data/animations.ts` merges generated metadata into `ANIMATION_REGISTRY` using `mergeGeneratedMeta()`.
3. `previewImage` is set from generated `meta.thumbnailSrc` when present.
4. `lib/gallery-adapter.ts` maps animation data to gallery card payloads (`image`, `title`, `tags`, etc.).
5. `src/templates/gallery/GalleryMainTemplate.client.tsx` renders `Uploaded_AnimationGallery_v1` and modal interactions.
6. Thumbnail updates can be produced by `scripts/generate-thumbnails.js` or uploaded through `app/api/animations/thumbnail/route.ts`.

## 5) WordPress Taxonomy / Feed Flow
1. Blog index server route `app/blog/page.tsx` fetches categories/posts via `lib/wordpress.ts`.
2. Client route logic in `app/blog/BlogIndexClient.tsx` responds to URL query (`q`, `category`, `page`) and refetches via `lib/wp-client.ts`.
3. Taxonomy rendering uses `fetchCategories()`, `fetchPostsWithTotal()`, and helper `getPostCategories()`.
4. Single-post UI in `app/blog/post/BlogPostClient.tsx` reads `?slug=...`, fetches with `fetchPostBySlug()`, and derives related posts by taxonomy IDs.
5. Featured media is resolved through `lib/wp-media.ts`.
6. Separate MDX blog path exists at `app/blog/_[slug]/page.tsx` and reads files from `content/blog/posts` (directory currently absent).

## 6) Safe Changes vs Risky Changes

Safe changes:
- Content edits in `content/` that preserve existing keys/slugs.
- Adding registry rows with valid `templateId` and `contentKey` mappings.
- Template styling/markup changes that keep prop contracts stable.
- Updating docs, reports, and handoff artifacts.

Risky changes:
- Renaming routes or changing route segment structures.
- Changing `templateId` strings without mapping updates in template registries.
- Changing `contentKey` strings without updating `contentByKey` or dynamic lookup logic.
- Editing generated files manually (`*.generated.ts`) without re-running generators.
- Modifying blog route semantics (`/blog/post?slug=` vs slug-based routing) without migration plan.

## Source-of-Truth Assessment
- Routes: App Router files under `app/`, with registry controlling many page-level mappings.
- Registry: `src/data/page-registry.csv` (source), `src/data/pageRegistry.generated.ts` (runtime artifact).
- Template mapping: `src/templates/registry.ts` (+ uploaded fallback file).
- Content mapping: `src/content/registry.ts`.
- JSON/generated content: `src/data/animationCatalog.generated.ts`, `src/data/animationMeta.overrides.json`.
- Shared components: `components/`, `src/templates/`, `lib/`.
- Gallery data: `src/data/animations.ts`, `src/data/animationGallery.ts`.
- Thumbnail logic: `scripts/generate-thumbnails.js`, `app/api/animations/thumbnail/route.ts`, merge logic in `src/data/animations.ts`.
- WordPress feed logic: `lib/wp-client.ts`, `lib/wordpress.ts`, and blog route files under `app/blog/`.

## Ambiguities
- Canonical blog post route is `uncertain`:
- There is no `app/blog/[slug]/page.tsx`.
- Active runtime path appears to be `app/blog/post/page.tsx` with `?slug=` query.
- `app/blog/_[slug]/page.tsx` exists for MDX-based posts and may represent a parallel/non-primary system.
- `content/blog/posts` directory is currently absent, so MDX blog route behavior is effectively content-empty unless that directory is created.
- Registry vs hand-authored route precedence for some pages is `uncertain` and should be validated before structural changes.
